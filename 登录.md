增加# 登录认证方案

## 概述

本项目采用**基于JWT的双Token机制**实现用户认证，采用混合方案：
- **Access Token**: 纯JWT方案，不设置白名单和黑名单
- **Refresh Token**: 使用Redis白名单存储，支持多设备区分

该方案在保证安全性的同时，简化了Access Token的验证流程，提高了性能。

### 新方案核心特点

1. **Access Token 纯JWT化**: 不依赖Redis存储，验证时只解析JWT，验证签名和过期时间
2. **Refresh Token 白名单**: 使用Redis Hash结构存储，按用户ID和设备ID分组
3. **多设备支持**: 通过设备ID区分不同登录设备，支持单设备登出和查看登录设备列表

## 问题回答

### Q1: 为什么Access Token采用纯JWT方案，不设置白名单和黑名单？

**答：**

1. **性能考虑**：每次请求都查Redis会增加延迟，尤其在高并发场景下。纯JWT方案只需在服务端解析验证签名，无需网络IO。

2. **JWT特性**：JWT本身已包含过期时间验证，只要签名正确且未过期，即可认为有效。这已经满足了Access Token的安全需求。

3. **简化架构**：无需维护Token状态，减少Redis存储开销和复杂度。

4. **安全性权衡**：
   - Access Token 有效期短（15分钟），即使被截获，窗口期有限
   - 配合Refresh Token的白名单机制，可以快速吊销任何异常Token
   - 真正的敏感操作可以通过更严格的验证（如二次验证）

**为什么不需要黑名单？**
- JWT的无状态特性决定了其一旦签发就无法撤销
- 但通过短期有效期的设计，风险可控
- 如需紧急吊销，可通过吊销对应的Refresh Token来阻止续期

**为什么不需要白名单？**
- 白名单的目的是验证Token是否有效
- JWT的签名验证本身就保证了Token的真实性
- 过期时间验证保证了时效性
- 两者结合已足够保证Access Token的有效性

### Q2: 为什么Refresh Token采用白名单方案？

**答：**

1. **长期有效性**：Refresh Token有效期7天，必须能够被吊销
2. **续期控制**：Refresh Token是获取新Access Token的唯一凭证，必须严格控制
3. **多设备管理**：需要知道用户从哪些设备登录，便于管理
4. **安全防护**：白名单存储可以快速定位和吊销特定设备的Token

### Q3: 如何实现多设备区分？

**答：**

1. **设备标识**：每个Token生成时需要传入deviceID（设备唯一标识，如设备序列号、UUID等）
2. **按设备分组**：Refresh Token白名单按 `用户ID:设备ID` 分组存储
3. **设备列表维护**：通过Redis Set记录用户登录过的所有设备
4. **单设备登出**：可以根据用户ID和设备ID吊销特定设备的Token

### Q4: 为什么选择双Token而不是单Token？

**答：**

单Token（尤其是纯JWT）存在以下问题：

1. **无法主动吊销**：JWT是无状态的，签发后无法撤销
2. **无法实现登出**：用户点击退出后，Token仍然有效
3. **无法多设备管理**：无法控制其他设备的登录状态
4. **安全性差**：有效期长则风险大，有效期短则用户体验差

双Token方案通过分离短期凭证和长期凭证来解决这些问题：
- **Access Token（15分钟）**：短期有效，安全可控，即使泄露窗口期短
- **Refresh Token（7天）**：长期有效，用于续期，可吊销、可管理

### Q5: Access Token和Refresh Token的有效期分别设置为多少合适？

**答：**

**Access Token有效期选择**：
- 一般设置为15-30分钟
- 考虑因素：安全性 vs 用户体验
- 本项目设置为15分钟，在安全和性能间取得平衡

**Refresh Token有效期选择**：
- 一般设置为7-30天
- 考虑因素：安全性 vs 便利性
- 本项目设置为7天，平衡了安全性和用户体验

**如何确定具体数值**：
- 安全敏感场景：Access Token 5分钟，Refresh Token 1天
- 普通场景：Access Token 15-30分钟，Refresh Token 7-14天
- 记住密码场景：可适当延长

### Q6: Token泄露后如何处理？

**答：**

1. **Access Token泄露**：
   - 有效期短（15分钟），泄露风险窗口有限
   - 用户修改密码时可吊销所有Refresh Token
   - 异常检测时可封禁账号

2. **Refresh Token泄露**：
   - 立即通过白名单机制吊销
   - 清除该设备的所有Token
   - 可通过`RevokeDeviceToken`或`RevokeAllUserTokens`实现

3. **预防措施**：
   - 使用HTTPS传输
   - 敏感操作二次验证
   - 记录登录设备IP/位置异常检测

### Q7: 如何防止Token被窃取？

**答：**

1. **传输层安全**：
   - 全站HTTPS
   - 防止中间人攻击

2. **存储安全**：
   - Access Token存localStorage（有XSS风险）
   - Refresh Token存HttpOnly Cookie（防XSS）
   - 避免存在localStorage中

3. **业务层安全**：
   - 敏感操作要求重新输入密码
   - 异常IP/设备检测
   - 频繁失败后限流

4. **技术方案**：
   - 签名算法使用HMAC-SHA256
   - 密钥足够复杂且定期轮换
   - Token payload避免敏感信息

### Q8: 如何处理Token续期时的并发请求？

**答：**

**问题场景**：当Access Token过期，多个请求同时发起刷新请求

**解决方案**：

1. **前端方案**（本项目采用）：
   ```javascript
   let isRefreshing = false;
   let refreshSubscribers = [];

   // 订阅刷新完成通知
   function subscribeTokenRefresh(callback) {
     refreshSubscribers.push(callback);
   }

   // 通知所有等待者
   function onTokenRefreshed(newToken) {
     refreshSubscribers.forEach(callback => callback(newToken));
     refreshSubscribers = [];
   }

   // 401处理
   if (error.response?.status === 401 && !isRefreshing) {
     isRefreshing = true;
     // 发起刷新请求
     const res = await refreshToken();
     isRefreshing = false;
     onTokenRefreshed(res.data.accessToken);
   } else if (isRefreshing) {
     // 加入等待队列
     return new Promise(resolve => {
       subscribeTokenRefresh(token => {
         error.config.headers.Authorization = `Bearer ${token}`;
         resolve(axios(error.config));
       });
     });
   }
   ```

2. **后端方案**：
   - 使用分布式锁防止并发刷新
   - 或者使用Token版本号机制

### Q9: 如何实现"异地登录踢人"功能？

**答：**

1. **实现思路**：
   - 每次登录生成新Token时，记录设备信息
   - 用户查看设备列表可以看到所有登录设备
   - 强制登出某一设备时，吊销该设备的Refresh Token白名单

2. **关键接口**：
   - `GetUserDevices`：获取用户所有登录设备
   - `RevokeDeviceToken`：吊销指定设备的Token
   - `RevokeAllUserTokens`：吊销所有设备Token

3. **实现代码**：
   ```go
   // 吊销指定设备
   func (s *TokenService) RevokeDeviceToken(ctx context.Context, userID int64, deviceID string) error {
       whitelistKey := RefreshWhitelistKey + fmt.Sprintf("%d:%s", userID, deviceID)
       // 获取该设备的所有Refresh Token并删除
       refreshTokenIDs, _ := s.rdb.HKeys(ctx, whitelistKey).Result()
       pipe := s.rdb.Pipeline()
       for _, rtID := range refreshTokenIDs {
           pipe.Del(ctx, RefreshTokenKey+rtID)
       }
       pipe.Del(ctx, whitelistKey)
       _, err := pipe.Exec(ctx)
       return err
   }
   ```

### Q10: 如果Redis集群故障，如何保证登录功能可用？

**答：**

1. **问题分析**：Refresh Token存在Redis中，Redis故障会导致无法刷新Token

2. **解决方案**：

   **方案A：本地缓存 + 降级**
   - 在Redis集群故障时，使用本地缓存的Refresh Token
   - 短暂降级：允许不验证Refresh Token直接续期
   - 记录故障期间的异常请求，后续审计

   **方案B：多级缓存**
   - 使用本地内存缓存 + Redis
   - 本地缓存作为Redis的备份
   - 定期同步到Redis

   **方案C：数据库降级**
   - 将Token信息存储到MySQL
   - Redis故障时从MySQL读取
   - 性能下降但保证可用

3. **本项目建议**：
   - 短期Access Token（15分钟）在Redis故障时不影响已登录用户
   - 提前预警：监控Redis连接数、内存使用率
   - 熔断降级：连续失败后返回友好提示

### Q11: 如何设计一个安全的登录接口？

**答：**

1. **验证码接口**（/user/code）
   - 限流：同一IP/手机号每分钟只能发送一次
   - 图形验证码：防止自动化脚本
   - 有效期短（5分钟）
   - 单次有效，使用后立即删除

2. **登录接口**（/user/login）
   - 验证码校验：一次性使用
   - 错误次数限制：连续5次错误后需图形验证码
   - 登录间隔限制：同一账号60秒内只能尝试一次
   - 密码错误锁定：连续5次错误后锁定15分钟

3. **日志审计**
   - 记录登录IP、地理位置、设备信息
   - 异常登录告警（异地登录、频繁失败）
   - 敏感操作日志

### Q12: JWT相比Session的优势和劣势？

**答：**

**优势**：

1. **无状态**：服务端不需要存储Token，分布式友好
2. **性能**：验证时不需要查询数据库/缓存
3. **跨域**：易于实现跨域API
4. **移动端友好**：Token机制更适合移动端

**劣势**：

1. **无法吊销**：签发后无法撤销（除非用黑名单）
2. **Payload膨胀**：Token包含越多数据，传输越大
3. **无法感知用户状态**：Token签发后无法强制失效

**适用场景**：
- 前后端分离架构
- 需要支持移动端
- 追求高性能
- 不需要实时下线用户

### Q13: Access Token过期时，Refresh Token策略如何选择？

**答：**

**本项目采用：主动刷新 + 被动刷新**

1. **主动刷新**（前端轮询）：
   ```javascript
   // 每分钟检查一次
   setInterval(() => {
     const remaining = expireTime - Date.now();
     if (remaining < 5 * 60 * 1000) {
       // 剩余5分钟时主动刷新
       refreshToken();
     }
   }, 60 * 1000);
   ```

2. **被动刷新**（收到401后）：
   - 前端拦截401错误
   - 自动使用Refresh Token换发新Token
   - 重试失败的请求

3. **续期策略选择**：
   - **滑动窗口**：每次请求都重置过期时间（用户体验好，安全性较低）
   - **固定过期**：Token到固定时间过期（安全性高，用户需重新登录）
   - **混合策略**：快过期时续期，如已超过总时长一半

### Q14: 如何保证Token的安全性？

**答：**

1. **签名算法**：使用HMAC-SHA256，避免使用HS256（对称加密，有泄露风险）

2. **密钥管理**：
   - 密钥足够复杂（256位）
   - 定期轮换（建议每3个月）
   - 使用密钥管理服务（KMS）

3. **Payload设计**：
   - 只包含必要信息（userID、deviceID）
   - 不包含敏感信息（密码、身份证）
   - 加入签发时间、版本号

4. **传输安全**：全程HTTPS

5. **存储安全**：
   - Access Token：localStorage（防XSS可存memory）
   - Refresh Token：HttpOnly Cookie（防XSS）

### Q15: 如何实现Token的自动续期？

**答：**

1. **本项目方案**：前端主动刷新 + 被动刷新

2. **后端滑动窗口续期**（可选方案）：
   ```go
   func (s *TokenService) ValidateAccessToken(ctx context.Context, tokenString string) (*AccessTokenInfo, error) {
       // 验证Token...
       // 验证通过后，滑动窗口续期：重置Redis中的过期时间
       if s.rdb.Exists(ctx, accessKey) {
           s.rdb.Expire(ctx, accessKey, AccessTokenTTL)
       }
       return tokenInfo, nil
   }
   ```
   注意：本项目的Access Token是纯JWT，不需要Redis续期。

3. **续期触发时机**：
   - 前端定时检查：剩余有效期不足时主动刷新
   - 每次请求响应中返回剩余时间，前端判断

### Q16: 分布式环境下的Token管理？

**答：**

1. **本项目架构**：
   - Access Token：纯JWT，天然支持分布式
   - Refresh Token：存储在Redis，支持集群

2. **Redis集群部署**：
   - 使用Redis Cluster或Sentinel
   - 保证高可用

3. **Token验证**：
   - 方式一：集中验证（所有请求都调用认证服务）
   - 方式二：分散验证（各服务自行验证JWT）
   - 本项目采用方式二，性能更好

4. **注意事项**：
   - 多实例部署时，密钥必须一致
   - 吊销Token需要保证Redis集群数据同步

### Q17: 前端如何安全存储Token？

**答：**

1. **存储方案对比**：

   | 方案 | 优点 | 缺点 | 适用场景 |
   |------|------|------|----------|
   | localStorage | 持久化，容量大 | 有XSS风险 | 非敏感场景 |
   | sessionStorage | 页面关闭清除 | 不持久 | 临时场景 |
   | HttpOnly Cookie | 防XSS | 有CSRF风险 | 安全敏感 |
   | 内存变量 | 最安全 | 不持久 | 敏感操作 |

2. **本项目方案**：
   - Access Token：localStorage
   - Refresh Token：HttpOnly Cookie

3. **安全建议**：
   - 敏感操作使用内存中的Token
   - 定期清理过期Token
   - 登录态检测到异常时主动清除

### Q18: 如果前端Token被XSS攻击窃取怎么办？

**答：**

1. **攻击原理**：
   - 攻击者注入恶意脚本
   - 读取localStorage中的Token
   - 使用窃取的Token冒充用户

2. **防御措施**：
   - **HttpOnly Cookie存储Token**（本项目采用）：
     - JavaScript无法读取
     - 完全防止XSS导致的Token泄露
   
   - **设置Token有效期更短**：
     - Access Token 15分钟
     - 减少泄露窗口

   - **IP/User-Agent检测**：
     - 检测异常IP/设备
     - 发现异常自动吊销

3. **应急响应**：
   - 清除所有登录态（修改密码）
   - 检查并修复XSS漏洞
   - 分析日志确定攻击范围

### Q19: 为什么使用JWT而不使用OAuth2.0？

**答：**

1. **OAuth2.0适用场景**：
   - 第三方登录
   - 授权访问其他服务
   - 复杂的权限管理

2. **本项目场景**：
   - 自有系统用户登录
   - 简单的用户认证
   - 不需要授权第三方

3. **JWT vs OAuth2.0**：
   - JWT是令牌格式标准
   - OAuth2.0是授权框架
   - 可以结合使用（JWT作为OAuth2.0的令牌格式）

### Q20: 生产环境如何监控Token相关指标？

**答：**

1. **关键指标**：

   | 指标 | 描述 | 告警阈值 |
   |------|------|----------|
   | 登录成功率 | 登录请求中成功的比例 | < 80% |
   | Token刷新失败率 | 刷新请求中失败的比例 | > 10% |
   | 活跃设备数 | 用户平均登录设备数 | 异常增长 |
   | 异地登录 | 跨地域的登录请求 | 检测到即告警 |
   | Token过期错误 | 401错误频率 | 突然增长 |

2. **监控方案**：
   - 使用Prometheus + Grafana
   - 埋点统计各接口调用量、耗时、错误率
   - 配置告警规则

3. **日志记录**：
   - 登录/登出日志
   - Token刷新日志
   - 异常登录告警

```
         │<──────────────────────────────────────────────
```

### 整体架构流程详细文字描述

1. **用户登录请求阶段**
   - 用户在前端界面输入手机号和获取到的验证码，点击登录按钮
   - 前端发起登录请求，携带 deviceId（设备唯一标识）
   - user-service 接收到登录请求后，验证手机号对应的验证码
   - 验证码验证通过后，查询或创建用户记录
   - 生成双Token对，Access Token为纯JWT，Refresh Token存入Redis白名单
   - 将Refresh Token设置到HttpOnly Cookie中
   - 返回Access Token和用户信息给前端

2. **后续请求鉴权阶段**
   - 前端在请求Header中携带Access Token
   - content-service的AuthMiddleware拦截请求
   - 直接解析JWT验证签名和过期时间（纯JWT验证，不查Redis）
   - 验证通过后执行业务逻辑
   - 返回业务数据给前端

3. **Token无感刷新阶段**
   - 当Access Token过期，前端收到401错误
   - 前端自动发起刷新请求，携带Refresh Token
   - user-service验证Refresh Token白名单
   - 验证通过后，吊销旧Token，生成新Token
   - 返回新Access Token给前端
   - 前端重试失败的请求

## 双Token机制详解

### 为什么需要双Token？

**单JWT问题**：
- 无法主动吊销
- 无法实现登出
- 无法多设备管理

**双Token方案优势**：
- Access Token短期有效，安全可控
- Refresh Token长期有效，支持续期
- 通过Refresh Token白名单实现吊销和多设备管理

### 双Token设计对比

| Token类型 | 有效期 | 存储方式 | 验证方式 | 用途 |
|-----------|--------|----------|----------|------|
| Access Token | 15分钟 | 前端localStorage (JWT) | 纯JWT解析，不查Redis | 访问需要认证的接口 |
| Refresh Token | 7天 | Redis白名单 + HttpOnly Cookie | 查Redis白名单 | 换发新的Access Token |

## Redis存储结构详解

### 完整介绍

本系统的Redis存储结构设计围绕新的双Token认证机制，采用混合存储方案。

### 设计背景

1. **Access Token纯JWT化**：不需要在Redis中存储Access Token状态，减少IO开销
2. **Refresh Token白名单**：需要支持多设备管理和快速吊销
3. **设备维度管理**：按用户ID和设备ID组织存储，便于多设备管理

### 核心数据结构

```go
// 常量定义
RefreshTokenKey     = "token:refresh:"      // Refresh Token 存储前缀
RefreshWhitelistKey = "token:whitelist:"    // Refresh Token 白名单前缀
UserDevicesKey      = "user:devices:"       // 用户设备列表前缀

AccessTokenTTL      = 15 * time.Minute     // Access Token 有效期
RefreshTokenTTL     = 7 * 24 * time.Hour   // Refresh Token 有效期
```

### 1. Refresh Token 存储（白名单）

```
Key:    token:refresh:<refreshTokenId>
Type:   String
Value:  userID:deviceID (如 "12345:iphone15-pro")
TTL:    7天
```

**作用**：验证Refresh Token是否有效

**示例**：
```
redis> GET token:refresh:x1y2z3w4v5u6t7s8r9q0p1o2n3m4l5k6j7i8h9g0f1e2d3c4b5a6
"12345:iphone15-pro"

redis> TTL token:refresh:x1y2z3...
(integer) 604800
```

### 2. Refresh Token 白名单（按设备分组）

```
Key:    token:whitelist:<userID>:<deviceID>
Type:   Hash
Fields: refreshTokenID -> 过期时间
TTL:    7天
```

**作用**：按设备管理Refresh Token，支持多设备区分

**示例**：
```
redis> HGETALL token:whitelist:12345:iphone15-pro
1) "x1y2z3..."
2) "2026-04-12T10:00:00Z"

redis> HGETALL token:whitelist:12345:huawei-p50
1) "a1b2c3..."
2) "2026-04-11T15:30:00Z"
```

### 3. 用户设备列表

```
Key:    user:devices:<userId>
Type:   Set
Members: deviceID 列表
TTL:    7天
```

**作用**：记录用户登录过的所有设备，用于多设备管理

**示例**：
```
redis> SMEMBERS user:devices:12345
1) "iphone15-pro"
2) "huawei-p50"
3) "xiaomi-13"

redis> SCARD user:devices:12345
(integer) 3
```

### 完整存储示意图

```
┌─────────────────────────────────────────────────────────────────┐
│                     Redis 存储结构总览                           │
└─────────────────────────────────────────────────────────────────┘

┌────────────────────────────────────────┐
│ token:refresh:refreshTokenID_1         │
│ (String, 7天)                          │
├────────────────────────────────────────┤
│ Value = "12345:iphone15-pro"           │
└────────────────────────────────────────┘
                    │
                    │ 映射到
                    ▼

└────────────────────────────────────────┘
```

## 代码实现要点

### 1. 生成双Token（GenerateTokenPair）

```go
// user-service/utils/token.go
func (s *TokenService) GenerateTokenPair(ctx context.Context, userID int64, deviceID string) (*TokenPair, error) {
    // 1. 生成Access Token ID
    accessTokenID := generateRandomToken(AccessTokenLength)

    // 2. 生成Access Token (纯JWT，不存Redis)
    accessToken, err := s.generateJWT(userID, deviceID, accessTokenID, AccessTokenTTL)
    if err != nil {
        return nil, fmt.Errorf("生成access token失败: %w", err)
    }

    // 3. 生成Refresh Token ID
    refreshTokenID := generateRandomToken(RefreshTokenLength)

    // 4. 写入Redis白名单
    pipe := s.rdb.Pipeline()

    // 白名单Hash: token:whitelist:<userID>:<deviceID>
    whitelistKey := RefreshWhitelistKey + fmt.Sprintf("%d:%s", userID, deviceID)
    pipe.HSet(ctx, whitelistKey, refreshTokenID, time.Now().Add(RefreshTokenTTL).Format(time.RFC3339))
    pipe.Expire(ctx, whitelistKey, RefreshTokenTTL)

    // 简单String: token:refresh:<refreshTokenID>
    refreshKey := RefreshTokenKey + refreshTokenID
    pipe.Set(ctx, refreshKey, fmt.Sprintf("%d:%s", userID, deviceID), RefreshTokenTTL)

    // 用户设备列表
    userDevicesKey := UserDevicesKey + fmt.Sprintf("%d", userID)
    pipe.SAdd(ctx, userDevicesKey, deviceID)
    pipe.Expire(ctx, userDevicesKey, RefreshTokenTTL)

    _, err = pipe.Exec(ctx)
    // ...
}
```

### 2. 验证Access Token（纯JWT验证）

```go
// user-service/utils/token.go
func (s *TokenService) ValidateAccessToken(ctx context.Context, tokenString string) (*AccessTokenInfo, error) {
    if tokenString == "" {
        return nil, fmt.Errorf("token不能为空")
    }

    // 纯JWT解析，不查Redis！
    token, err := jwt.ParseWithClaims(tokenString, &JWTClaims{}, func(token *jwt.Token) (interface{}, error) {
        if _, ok := token.Method.(*jwt.SigningMethodHMAC); !ok {
            return nil, fmt.Errorf("unexpected signing method: %v", token.Header["alg"])
        }
        return []byte(TokenSecret), nil
    })

    if err != nil {
        return nil, fmt.Errorf("token解析失败: %w", err)
    }

    if !token.Valid {
        return nil, fmt.Errorf("token无效")
    }

    claims, ok := token.Claims.(*JWTClaims)
    if !ok {
        return nil, fmt.Errorf("token claims无效")
    }

    return &AccessTokenInfo{
        UserID:    claims.UserID,
        TokenID:   claims.TokenID,
        DeviceID:  claims.DeviceID,
        CreatedAt: claims.IssuedAt.Time,
        ExpireAt:  claims.ExpiresAt.Time,
    }, nil
}
```

### 3. 刷新Token（RefreshToken）

```go
// user-service/utils/token.go
func (s *TokenService) RefreshToken(ctx context.Context, refreshTokenID string, deviceID string) (*TokenPair, error) {
    // 1. 验证Refresh Token存在 (查白名单)
    refreshKey := RefreshTokenKey + refreshTokenID
    value, err := s.rdb.Get(ctx, refreshKey).Result()
    if err == redis.Nil {
        return nil, fmt.Errorf("refresh token已过期或无效")
    }
    if err != nil {
        return nil, fmt.Errorf("查询refresh token失败: %w", err)
    }

    // 2. 解析 userID:deviceID
    var userID int64
    var originalDeviceID string
    fmt.Sscanf(value, "%d:%s", &userID, &originalDeviceID)

    // 3. 吊销旧的Refresh Token
    s.rdb.Del(ctx, refreshKey)
    whitelistKey := RefreshWhitelistKey + fmt.Sprintf("%d:%s", userID, originalDeviceID)
    s.rdb.HDel(ctx, whitelistKey, refreshTokenID)

    // 4. 生成新的双Token
    return s.GenerateTokenPair(ctx, userID, deviceID)
}
```

```javascript
    config.headers.Authorization = `Bearer ${token}`;
  }
  return config;
});

// 响应拦截 - 处理Token过期
axios.interceptors.response.use(
  response => response,
  async error => {
    if (error.response?.status === 401) {
      const refreshToken = localStorage.getItem('refreshToken');
      if (refreshToken) {
        try {
          const res = await axios.post('/user/refresh', { refreshToken });
          if (res.data.code === 200) {
            localStorage.setItem('accessToken', res.data.data.accessToken);
            localStorage.setItem('refreshToken', res.data.data.refreshToken);
            localStorage.setItem('tokenExpire', Date.now() + res.data.data.expiresIn * 1000);
            error.config.headers.Authorization = `Bearer ${res.data.data.accessToken}`;
            return axios(error.config);
          }
        } catch (e) {
          console.error('刷新Token失败', e);
        }
      }
      // 刷新失败，清除登录状态
      localStorage.clear();
      window.location.href = '/login';
    }
    return Promise.reject(error);
  }
);
```

### 定时刷新（可选）

```javascript
// 每分钟检查一次
setInterval(async () => {
  const expireTime = localStorage.getItem('tokenExpire');
  const remaining = expireTime - Date.now();

  // 剩余不足5分钟时刷新
  if (remaining > 0 && remaining < 5 * 60 * 1000) {
    const refreshToken = localStorage.getItem('refreshToken');
    try {
      const res = await axios.post('/user/refresh', { refreshToken });
      localStorage.setItem('accessToken', res.data.data.accessToken);
      localStorage.setItem('refreshToken', res.data.data.refreshToken);
    } catch (e) {
      console.error('Token刷新失败', e);
    }
  }
}, 60 * 1000);
```

## 与旧方案对比

| 特性 | 旧方案 | 新方案 |
|------|--------|--------|
| Access Token存储 | JWT + Redis Set | 纯JWT |
| Access Token验证 | 解析JWT + 查Redis | 纯JWT解析 |
| Refresh Token存储 | Redis String | Redis Hash (白名单) |
| 多设备支持 | 通过user:tokens Set | 通过user:devices Set + 设备维度白名单 |
| 性能 | 每次验证2次Redis | 验证0次Redis |

### 新方案优势

1. **更高性能**：Access Token验证不查Redis，减少网络IO
2. **更简单**：架构更简单，不需要维护Token状态
3. **更好管理**：设备维度管理，支持单设备登出
4. **安全可控**：短期Access Token + 可吊销的Refresh Token

### content-service调用user-service

```go
// content-service/utils/auth.go
func validateToken(ctx context.Context, token string) (*model.UserDTO, error) {
    req, _ := http.NewRequestWithContext(ctx, "POST", "http://localhost:8081/user/validate", nil)
    req.Header.Set("Authorization", "Bearer "+token)

    client := &http.Client{}
    resp, err := client.Do(req)
    if err != nil {
        return nil, err
    }
    defer resp.Body.Close()

    var result struct {
        Code    int            `json:"code"`
        Message string         `json:"message"`
        Data    *model.UserDTO `json:"data"`
    }
    json.NewDecoder(resp.Body).Decode(&result)

    if result.Code != 200 {
        return nil, errors.New(result.Message)
    }

    return result.Data, nil
}
```
