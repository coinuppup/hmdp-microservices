### 2.0 Etcd深度原理

#### 2.0.1 Etcd是什么？

Etcd是一个分布式的、可靠的键值存储系统，用于存储分布式系统的关键数据。CoreOS公司开发，现已被Kubernetes广泛采用。

```
┌─────────────────────────────────────────────────────────────────────────┐

├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│    ┌──────────────┐     ┌──────────────┐     ┌──────────────┐          │
│    │  应用服务A   │     │  应用服务B   │     │  应用服务C   │          │
│    └──────┬───────┘     └──────┬───────┘     └──────┬───────┘          │
│           │                    │                    │                   │
│           └────────────────────┼────────────────────┘                   │
│                                │                                        │
│                    ┌───────────▼───────────┐                           │
│                    │       Etcd            │                           │
│                    │   (配置中心/注册中心)  │                           │
│                    │                       │                           │
│                    │  - 服务注册与发现     │                           │
│                    │  - 配置管理           │                           │

│                    └───────────────────────┘                           │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

#### 2.0.2 Raft协议原理

Etcd使用Raft协议保证分布式一致性。Raft是一种共识算法，核心是将一致性分解为多个子问题：

```
┌─────────────────────────────────────────────────────────────────────────┐
│                         Raft协议核心概念                                 │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  Raft中的三种角色：                                                     │
│                                                                         │
│  ┌─────────────┐     ┌─────────────┐     ┌─────────────┐              │
│  │   Leader    │     │  Follower   │     │ Candidate  │              │
│  │   领导者    │     │   跟随者    │     │   候选者    │              │
│  ├─────────────┤     ├─────────────┤     ├─────────────┤              │
│  │ 处理所有    │     │ 响应Leader │     │ 发起选举    │              │
│  │ 写请求      │     │ 的请求      │     │ 投票请求    │              │
│  │ 负责复制    │     │ 参与投票    │     │             │              │
│  │ 数据到      │     │             │     │             │              │
│  │ Follower   │     │             │     │             │              │
│  └─────────────┘     └─────────────┘     └─────────────┘              │
│                                                                         │
│  角色转换关系：                                                          │
│                                                                         │
│      ┌──────────┐      赢得选举       ┌──────────┐                    │
│      │Candidate │ ──────────────────▶ │  Leader  │                    │
│      └──────────┘                     └──────────┘                    │
│           ▲                                │                          │
│           │                                │                          │
│           │      收到Leader心跳           │                          │
│           └───────────────────────────────┘                          │
│           │                                ▲                          │
│           │                                │                          │
│      选举超时         ┌──────────┐        │  选举超时               │
│      没有收到心跳     │ Follower │ ◀──────┘  没有收到心跳           │
│                       └──────────┘                                  │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

**Raft协议核心机制**：

```
┌─────────────────────────────────────────────────────────────────────────┐
│                         Raft协议工作流程                                 │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  1. 领导者选举                                                           │
│     - 集群启动时，所有节点都是Follower                                   │
│     - 选举超时（150-300ms随机）后，节点变为Candidate                    │
│     - Candidate发起选举，给自己投票，向其他节点请求投票                  │
│     - 获得多数派投票的节点成为Leader                                     │
│                                                                         │
│  2. 日志复制                                                             │
│     - Client发送写请求到Leader                                          │
│     - Leader将操作追加到本地日志                                        │
│     - Leader并行发送AppendEntries到所有Follower                         │
│     - Follower收到后追加到本地日志，返回成功                             │
│     - Leader收到多数派成功后，提交日志，应用到状态机                     │
│     - Leader在下一次心跳中通知Follower提交                              │
│                                                                         │
│  3. 任期（Term）                                                         │
│     - Term是逻辑时钟，每个任期最多一个Leader                             │
│     - Term单调递增，用于识别过期信息                                     │
│     - 如果Follower收到更高Term的请求，自动成为Follower                  │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

**WAL（Write Ahead Log）预写日志**：

```
┌─────────────────────────────────────────────────────────────────────────┐
│                         WAL日志机制                                      │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  为什么需要WAL？                                                        │
│  - 数据先写内存，再写磁盘                                                │
│  - 如果在写磁盘前崩溃，数据会丢失                                        │
│  - WAL：所有操作先记录日志，持久化后再执行                               │
│                                                                         │
│  流程：                                                                  │
│  1. 收到写请求                                                           │
│  2. 写入WAL日志（持久化到磁盘）                                         │
│  3. 写入内存（MVCC）                                                    │
│  4. 返回客户端成功                                                       │
│                                                                         │
│  好处：                                                                  │
│  - 即使崩溃也能通过日志恢复数据                                          │
│  - 批量写入提高性能                                                      │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

**Snapshot快照机制**：

```
┌─────────────────────────────────────────────────────────────────────────┐
│                         Snapshot快照机制                                 │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  问题：WAL日志会无限增长，磁盘会满                                       │
│                                                                         │
│  解决：定期做快照，清理旧的WAL                                           │
│                                                                         │
│  流程：                                                                  │
│  1. 定期（如每10000次操作）触发快照                                      │
│  2. 将当前状态机数据写入快照文件                                         │
│  3. 清理之前的WAL日志                                                    │
│  4. 节点启动时，先加载快照，再重放WAL                                    │
│                                                                         │
│  示例：                                                                  │
│  - 初始状态：空                                                          │
│  - 写入：k1=v1, k2=v2, k3=v3                                            │
│  - 快照：保存 {k1:v1, k2:v2, k3:v3}                                     │
│  - 写入：k4=v4                                                          │
│  - 重放：从快照恢复 + 重放k4=v4                                         │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

#### 2.0.5 Etcd集群脑裂问题

```
┌─────────────────────────────────────────────────────────────────────────┐
│                         脑裂问题及解决                                   │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  什么是脑裂？                                                            │
│  集群由于网络故障分裂成多个部分，每部分都认为自己是主节点                │
│                                                                         │
│  场景：                                                                  │
│  ┌─────────────────────────────────────────────────────────────────┐   │
│  │     集群3节点                                                     │   │
│  │                                                              │   │
│  │    Node1        Node2        Node3                          │   │
│  │    (Leader)     (Follower)   (Follower)                     │   │
│  │                                                              │   │
│  │         网络分区                                              │   │
│  │              ✘                                                │   │
│  │    ┌──────────┴──────────┐                                   │   │
│  │    │                     │                                   │   │
│  │    ▼                     ▼                                   │   │
│  │ Node1                   Node2+Node3                         │   │
│  │ (认为自己是Leader)      (重新选举Leader)                     │   │
│  │                                                              │   │
│  └─────────────────────────────────────────────────────────────────┘   │
│                                                                         │
│  Raft如何解决：                                                          │
│  1. 少数派无法获得多数派投票                                             │
│  2. 即使网络恢复，旧Leader发现更高Term，自动降级为Follower              │
│  3. 数据不会丢失或覆盖                                                  │
│                                                                         │
│  总结：Raft协议通过"多数派投票"和"Term机制"保证脑裂时数据安全            │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

### 2.2 服务注册原理

项目代码位置：`common/etcd/register.go`

```go
// ServiceRegister 服务注册结构体
type ServiceRegister struct {
    client        *clientv3.Client  // etcd客户端
    serviceName   string             // 服务名称
    serviceAddr   string             // 服务地址
    serviceTTL    int64              // 租约过期时间
    leaseID       clientv3.LeaseID   // 租约ID
    keepAliveChan <-chan *clientv3.LeaseKeepAliveResponse  // 心跳通道
}
```

**核心流程**：

```
┌─────────────────────────────────────────────────────────────────────────┐
│                         服务注册流程                                      │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  1. 创建Etcd客户端                                                     │
│     client, err := clientv3.New(clientv3.Config{                        │
│         Endpoints:   []string{"localhost:2379"},                       │
│         DialTimeout: 5 * time.Second,                                  │
│     })                                                                  │
│                                                                         │
│  2. 创建租约（TTL=10秒）                                               │
│     resp, err := client.Grant(ctx, 10)                                 │
│     leaseID = resp.ID                                                  │
│                                                                         │
│  3. 注册服务（绑定租约）                                               │
│     key := "/services/user-service/192.168.1.100:8081"                 │
│     value := "192.168.1.100:8081"                                      │
│     client.Put(ctx, key, value, clientv3.WithLease(leaseID))           │
│                                                                         │
│  4. 启动心跳续期                                                       │
│     keepAliveChan, err := client.KeepAlive(ctx, leaseID)               │
│     go handleKeepAlive()  // 持续续期                                   │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

### 2.3 服务发现原理

项目代码位置：`common/etcd/discovery.go`

```go
// ServiceDiscovery 服务发现结构体
type ServiceDiscovery struct {
    client       *clientv3.Client  // etcd客户端
    serviceName  string            // 服务名称
    serviceAddrs []string          // 服务地址列表
```


### 2.4 租约机制详解

**为什么需要租约？**

```
┌─────────────────────────────────────────────────────────────────────────┐
│                         租约机制的作用                                    │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  问题：如果服务崩溃，没有手动注销，                                           │
│       别的服务还会尝试调用它                                                │
│                                                                         │
│  解决：租约机制（Lease）                                                  │
│       - 服务注册时绑定租约（如10秒）                                       │
│       - 定期心跳续期（保持租约有效）                                        │
│       - 服务崩溃后，心跳停止，租约自动过期                                   │
│       - 过期后自动从etcd中删除                                            │
│                                                                         │
│  流程：                                                                  │
│  ┌──────────┐     租约10s    ┌──────────┐                              │
│  │ 服务A    │ ────────────▶  │   Etcd   │                              │
│  └──────────┘                └────┬─────┘                              │
│        │                          │                                     │
│        │  每3秒心跳续期           │                                     │
│        │─────────────────────────▶                                     │
│        │                          │                                     │
│        │       服务崩溃           │                                     │
│        │          ✘              │                                     │
│        │                          │                                     │
│        │                    10秒后租约过期                               │
│        │                          │                                     │
│        │              服务A从etcd中删除                                  │
│        │                          ▼                                     │
│        │                ┌──────────────┐                               │
│        │                │  服务B发现   │                               │
│        │                │  服务A不可用 │                               │
│        │                └──────────────┘                               │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

**代码实现**：

```go
// 创建租约
resp, _ := client.Grant(ctx, 10)  // 10秒过期

// 绑定租约注册服务
client.Put(ctx, key, value, clientv3.WithLease(resp.ID))

// 启动心跳续期
keepAlive, _ := client.KeepAlive(ctx, resp.ID)

// 处理心跳响应
go func() {
    for range keepAlive {
        // 正常续期，无需操作
    }
}()
```

### 4.3 压测参数设计

```
┌─────────────────────────────────────────────────────────────────────────┐
│                         压测参数详解                                      │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  为什么这样设计压测参数？（面试必问）                                    │
│                                                                         │
│  1. 并发数设计（400连接）                                               │
│     ┌─────────────────────────────────────────────────────────────────┐ │

│     │                                                                 │ │
│     │ 计算依据：                                                      │ │
│     │ - 单个HTTP连接每秒可处理 10-20 个请求                          │ │
│     │ - 400连接 × 15请求/秒 = 6000 QPS                               │ │
│     │ - 考虑到网络开销和服务器处理能力，400是合理的上限             │ │
│     │                                                                 │ │
│     │ 实际考量：                                                      │ │
│     │ - 过高并发会导致测试客户端成为瓶颈                              │ │
│     │ - 真实场景中，流量会通过Nginx负载均衡                          │ │
│     │ - 400连接足以模拟单机最大负载                                  │ │
│     └─────────────────────────────────────────────────────────────────┘ │
│                                                                         │
│  2. 线程数设计（12线程）                                                │
│     ┌─────────────────────────────────────────────────────────────────┐ │
│     │ 思考：线程数与连接数的关系                                     │ │
│     │                                                                 │ │
│     │ 计算依据：                                                      │ │
│     │ - CPU核心数通常为4-16核                                        │ │
│     │ - wrk每个线程处理一批连接                                      │ │
│     │ - 12线程可以充分利用多核CPU                                    │ │
│     │                                                                 │ │
│     │ 经验值：                                                        │ │
│     │ - 线程数 = CPU核心数 * 2 是个不错的起点                        │ │
│     │ - 过多线程会增加上下文切换开销                                 │ │
│     └─────────────────────────────────────────────────────────────────┘ │
│                                                                         │
│  3. 压测时长设计（30秒）                                                │
│     ┌─────────────────────────────────────────────────────────────────┐ │
│     │ 思考：时间过长或过短有什么问题？                                │ │
│     │                                                                 │ │
│     │ 时长过短（<10秒）：                                            │ │
│     │ - 预热阶段数据影响结果准确性                                    │ │
│     │ - 无法观察到持续压力下的表现                                    │ │
│     │                                                                 │ │
│     │ 时长过长（>2分钟）：                                           │ │
│     │ - 占用测试资源，影响效率                                        │ │
│     │ - 秒杀场景库存耗尽后无意义                                     │ │
│     │                                                                 │ │
│     │ 30秒是个平衡点：                                                │ │
│     │ - 足够完成预热和稳定压测                                        │ │
│     │ - 观察持续压力下的表现                                          │ │
│     │ - 不会让测试等待太久                                            │ │
│     └─────────────────────────────────────────────────────────────────┘ │
│                                                                         │
│  4. 库存数量设计（1000件）                                              │
│     ┌─────────────────────────────────────────────────────────────────┐ │
│     │ 思考：为什么不是1件或10000件？                                 │ │
│     │                                                                 │ │
│     │ 库存太少（1件）：                                               │ │
│     │ - 并发进来全部失败，无法验证超卖逻辑                           │ │
│     │ - 1件商品无法模拟真实秒杀场景                                   │ │
│     │                                                                 │ │
│     │ 库存太多（10000件）：                                           │ │
│     │ - 所有请求都能成功，无法验证高并发扣减                         │ │
│     │ - 压测时间会很长                                                │ │
│     │                                                                 │ │
│     │ 1000件的合理性：                                                │ │
│     │ - 足够验证超卖逻辑（1000个订单 vs 5000个请求）                 │ │
│     │ - 能区分出正常扣减和超卖                                        │ │
│     │ - 压测时间适中                                                  │ │
│     └─────────────────────────────────────────────────────────────────┘ │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

服务间通信使用gRPC，外部请求通过HTTP，内部服务调用通过gRPC。服务注册与发现使用Etcd，消息队列使用Kafka实现异步处理和削峰。

### 问题2：为什么选择Etcd作为服务注册中心？

**口述回答**：

选择Etcd主要考虑以下几点：

1. **语言契合**：项目使用Go语言开发，Etcd也是Go语言编写，客户端支持好。
2. **一致性保证**：Etcd使用Raft协议，保证强一致性，服务发现准确。
3. **租约机制**：Etcd支持租约（Lease）机制，服务注册后需要心跳续期，如果服务崩溃，心跳停止，租约自动过期，服务自动从列表中移除。这个机制非常适合服务注册场景。
4. **高性能**：Etcd使用Go实现，性能高，延迟低，适合生产环境。
5. **生态兼容**：与Kubernetes原生支持，后续如果容器化部署可以无缝集成。

---

### 问题3：Etcd的租约机制是如何工作的？

**口述回答**：

Etcd的租约机制是服务注册的核心：

1. **创建租约**：服务启动时向Etcd申请一个租约，比如设置TTL为10秒。
2. **绑定服务**：将服务地址绑定到这个租约上注册到Etcd。
3. **心跳续期**：服务启动一个后台协程，每隔TTL/3时间（比如3秒）向Etcd发送心跳，证明服务还活着。
4. **自动过期**：如果服务崩溃，心跳停止，10秒后租约过期，Etcd自动删除该服务节点。
5. **服务发现**：其他服务通过监听Etcd的前缀（/services/服务名/），实时感知服务节点的变化。

这个机制保证了：当服务崩溃时，能够及时从注册中心移除，避免调用到已经不可用的服务。

### 问题4：项目中的压测结果是什么？做了哪些优化？

**口述回答**：

我们的压测结果是：秒杀接口支持 **5000+QPS**，**零超卖**，订单创建成功率 **100%**，接口平均延时 **低于50ms**。

主要优化手段有五个方面：

**第一，Redis Lua脚本实现原子操作。** 我使用Lua脚本在Redis端原子执行一人一单检查和库存扣减。为什么要用Lua？因为高并发场景下，如果先检查库存、再扣减库存，这两个操作不是原子的，会产生竞态条件，可能导致超卖。而Lua脚本在Redis中是单线程执行的，整个脚本作为原子操作执行，中间不会被其他请求打断，从根本上避免了并发问题。

**第二，库存预扣减。** 所有请求先在Redis层扣减库存，只有扣减成功的请求才会继续处理。这样可以拦截99%以上的流量，只有很少一部分请求会打到数据库，大大减轻了数据库压力。

**第三，Kafka异步下单。** 传统的做法是同步操作数据库，但数据库磁盘IO是瓶颈，高并发下会拖垮整个系统。我将订单创建完全异步化，用户请求到达后，Redis扣减库存成功就立即返回，订单的数据库写入通过Kafka异步处理。这样做的好处是：主流程非常快，用户不用等待；同时Kafka起到削峰填谷的作用，把瞬时高并发转化为平稳的后台处理。

**第四，多层防护杜绝超卖。** 除了Redis的原子操作，我在数据库层面还加了乐观锁，SQL语句是 `WHERE stock > 0`，只有库存大于0才能扣减成功；同时订单表有联合唯一索引，防止同一用户重复下单。即使Redis层出现问题，数据库层也能保证数据一致性。

**第五，分布式锁防止缓存击穿。** 秒杀活动开始时，大量请求会同时查询数据库，如果缓存不存在的话，可能导致数据库被打挂。我使用了Redis分布式锁，保证同一时间只有一个请求去查询数据库，其他请求等待锁释放后直接从缓存获取。

### 问题5：秒杀系统中如何保证不超卖？

**口述回答**：

超卖问题的本质是并发请求下的竞态条件。我通过三层防护来保证零超卖：

**第一层：Redis Lua脚本原子操作**

这是最核心的防护。我把库存检查和扣减封装在一个Lua脚本里：
```lua
local stock = redis.call('get', KEYS[1])
if not stock or tonumber(stock) <= 0 then
    return -1
end
return redis.call('decr', KEYS[1])
```

为什么要这样做？传统做法是先GET查询库存，再DECR扣减，这两个操作之间有时间窗口，其他请求可能在这个窗口内也查询到库存充足，然后都去扣减，导致超卖。而Lua脚本在Redis中是单线程执行的，整个检查+扣减过程作为一个原子操作完成，中间不会被其他请求打断，所以不会出现超卖。

**第二层：数据库乐观锁**

即使Redis层出了问题，还有数据库兜底。我的SQL是这样写的：
```sql
UPDATE tb_seckill_voucher SET stock = stock - 1
WHERE voucher_id = ? AND stock > 0
```

注意这里的 `AND stock > 0` 条件，只有库存大于0才能更新成功。如果影响行数为0，说明库存已经不足，这就是乐观锁的原理。相比悲观锁（SELECT FOR UPDATE），乐观锁不占用数据库连接，性能更好，更适合高并发场景。

**第三层：唯一索引**

订单表我对 user_id 和 voucher_id 建了联合唯一索引，保证同一个用户只能成功创建一张订单。即使前两层防护都失效，数据库也会因为唯一索引冲突而拒绝重复插入，这是最后一道防线。

总结一下为什么用三层：Redis Lua脚本拦截了99%的流量，性能最好；数据库乐观锁作为兜底，保证最终一致性；唯一索引防止重复订单。三层配合，既保证了高性能，又保证了数据安全。

### 问题6：项目中服务发现是如何实现的？

**口述回答**：

服务发现我使用的是Etcd，主要分为服务注册和服务发现两个部分：

**服务注册流程：**
每个服务启动时，会向Etcd写入一个Key-Value。Key的格式是 `/services/服务名/地址`，比如 `/services/user-service/192.168.1.100:8081`，Value就是服务地址。同时，我会给这个Key绑定一个租约（Lease），设置TTL为10秒。服务启动后，会定期向Etcd发送心跳续约，只要服务活着，租约就不会过期。如果服务崩溃了，心跳停止，10秒后租约过期，Etcd会自动删除这个服务节点。

**服务发现流程：**
服务消费者并不是每次调用都去查询Etcd，那样性能太差。我采用的是「监听+本地缓存」的方案：消费者启动时从Etcd获取一次完整的服务列表，缓存在本地内存中；同时启动一个goroutine长期监听Etcd的前缀（比如监听 `/services/user-service/`），当有服务节点新增或删除时，Etcd会通过Watch机制推送变更事件，消费者实时更新本地缓存。

**负载均衡：**
当需要调用服务时，从本地缓存的地址列表中，使用简单的轮询策略选择一个节点。

这样做的好处是：既保证了服务变化的实时感知（通过Watch），又保证了性能（通过本地缓存避免频繁查询Etcd）。即使Etcd暂时不可用，本地缓存的服务地址也能支撑一段时间，不影响正常调用。

### 问题7：项目中使用了哪些Redis数据结构？

**口述回答**：

我们项目根据不同业务场景，使用了Redis的八种数据结构：

**第一，String（字符串）**，这是最常用的。主要用于：验证码存储，比如 `login:code:手机号`，Value是6位数字；Token存储，用户登录后的token缓存；普通缓存值。

**第二，Hash（哈希）**，用于存储结构化数据。比如用户会话信息 `login:user:用户ID`，Hash的field是用户属性，Value是属性值；Token白名单也用Hash存储，方便按用户ID查询。

**第三，Set（集合）**，特点是成员唯一。我用在两个场景：关注列表 `follows:用户ID`，一个用户关注的所有人；秒杀一人一单 `seckill:order:优惠券ID`，存储已经购买的用户ID，Set的天然去重特性正好满足「一个用户只能买一次」的需求。

**第四，ZSet（有序集合）**，特点是成员唯一但可以按分数排序。用在博客点赞 `blog:liked:博客ID`，分数是点赞时间戳，这样按时间排序展示点赞列表；Feed流也是用ZSet，分数是发布时间，实现时间线功能。

**第五，BitMap（位图）**，适合存储大量布尔值。用在用户签到 `user:sign:年月:用户ID`，BitMap的每一位对应一天是否签到，31天只需要4字节内存，非常节省空间。

**第六，HyperLogLog**，用于统计海量数据的基数。比如UV统计 `uv:stats:日期`，统计每天有多少独立用户访问过。HyperLogLog只需要12KB内存就能统计2^64个用户，误差在0.81%左右，非常适合大数据量统计。

**第七，Geo（地理位置）**，用于附近商户功能。存储商户的经纬度坐标，可以通过GEORADIUS命令查询附近多少米内的商户。

**第八，Stream（流）**，这是Redis 5.0引入的数据结构，用于消息队列。我用 `stream:orders` 存储订单消息，比List类型更强大，支持消费者组、消息确认等特性。

选择数据结构的核心依据是业务场景：需要排序用ZSet，需要去重用Set，需要统计独立访问量用HyperLogLog，需要地理位置用Geo。

### 问题8：项目中MySQL使用了哪些索引？

**口述回答**：

我们项目主要使用了四种索引：

**第一，主键索引。** 每个表都有id作为主键，这是最基本的索引。主键索引的特点是唯一且非空，B+树结构，叶子节点存储完整的数据行。

**第二，唯一索引。** 有两个场景用了唯一索引：用户表的phone字段，保证手机号唯一，这是登录的依据；关注表的联合唯一索引（user_id + follow_user_id），保证一个用户只能关注同一个人一次，这个在业务上也是必须的。

**第三，普通索引。** 用于加速查询。比如博客表的user_id字段，用于查询某个用户发布的所有博客；评论表的blog_id字段，用于查询某个博客的所有评论；商铺表的type_id字段，用于按类型筛选商铺；优惠券表的shop_id字段，用于查询某个商铺的所有优惠券；订单表的user_id和voucher_id字段，用于查询用户的购买记录。

**第四，联合索引。** 这是我重点优化的地方。博客表有一个联合索引（liked DESC, create_time DESC），按点赞数降序、时间降序排列。这个索引是为了优化热门博客的排序查询。如果没有这个索引，查询热门博客需要全表扫描加filesort，非常慢；有了索引后，直接按索引顺序返回结果，性能提升几十倍。

索引设计的心得是：不要盲目加索引，要根据实际查询场景来设计。可以通过慢查询日志或者EXPLAIN分析SQL，找出查询瓶颈，针对性地添加索引。

### 追问2：服务注册中心挂了对系统有什么影响？

**回答**：

**1. 服务注册层面（影响程度：高）**
```
- 新启动的服务无法注册到注册中心
- 现有服务无法更新自己的地址信息
- 服务健康检查失效（无法上报心跳）
```

**2. 服务发现层面（影响程度：中）**
```
- 新启动的服务消费者无法获取服务列表
- 无法感知已有服务的上线/下线
- 但！已建立的长连接和本地缓存仍可正常使用
```

**5. 项目中的实现**

我们项目采用的就是方案1（本地缓存+定期同步）：

```go
// ServiceDiscovery 结构体
type ServiceDiscovery struct {
    client       *clientv3.Client
    serviceName  string
    serviceAddrs []string  // 本地缓存服务地址列表
}

// 获取服务地址（优先使用本地缓存）
func (d *ServiceDiscovery) GetServiceAddr() (string, error) {
    if len(d.serviceAddrs) == 0 {
        return "", fmt.Errorf("no service found")
    }
    // 使用本地缓存，即使Etcd挂了也能工作
    addr := d.serviceAddrs[0]
    d.serviceAddrs = append(d.serviceAddrs[1:], addr)
    return addr, nil
}

// 定期刷新服务列表（兜底机制）
func (d *ServiceDiscovery) refreshServiceList() error {
    // 从Etcd获取最新服务列表
    resp, err := d.client.Get(ctx, prefix, clientv3.WithPrefix())
    // 更新本地缓存
    d.serviceAddrs = parseAddrs(resp.Kvs)
    return nil
}
```

这种设计保证了：
- Etcd挂了后，本地缓存仍可支撑一段时间
- 已建立的连接不受影响
- 新启动的服务需要等待Etcd恢复

**总结**：
服务注册中心挂了，影响主要体现在"新服务无法注册"和"无法感知服务变化"，但已建立的连接和本地缓存可以继续工作一段时间。这是典型的AP系统设计，优先保证可用性，通过本地缓存和定期同步来提高容错能力。

**第二步：参数设计（核心！面试常问）**

我设置的是：**12线程 + 400并发 + 30秒 + 1000库存**

为什么这样设置？来算一笔账：

```
┌─────────────────────────────────────────────────────────────────┐
│                    参数计算过程                                  │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│ QPS目标：5000+                                                  │
│                                                                 │
│ 1. 并发数计算：                                                 │
│    单个HTTP连接每秒可处理 10-15 个请求                          │
│    400并发 × 15请求/秒 = 6000 QPS  （理论最大值）              │
│    400并发 × 12请求/秒 = 4800 QPS  （实际保守值）              │
│    → 400并发能覆盖5000 QPS目标                                  │
│                                                                 │
│ 2. 线程数计算：                                                 │
│    wrk是单线程事件循环，但可以通过多线程利用多核                │
│    线程数 = CPU核心数 × 2                                       │
│    8核 × 2 = 16，取12（留4核给系统）                           │
│    → 12线程能充分利用CPU                                       │
│                                                                 │
│ 3. 时长计算：                                                   │
│    时长太短（<10秒）：预热数据影响准确性                        │
│    时长太长（>2分钟）：库存耗尽后无意义                         │
│    30秒：足够预热 + 观察持续表现 + 等待时间可接受               │
│                                                                 │
│ 4. 库存计算：                                                   │
│    库存太少（100件）：并发进来全失败，无法验证超卖              │
│    库存太多（10000件）：全成功，无法验证高并发扣减              │
│    1000件：5000并发请求，能扣完，验证超卖逻辑                   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**参数汇总：**

| 参数 | 设置值 | 为什么这样设置 |
|------|--------|----------------|
| 并发数 | 400 | 400×15=6000，覆盖5000目标 |
| 线程数 | 12 | 8核×2=16，取12平衡开销 |
| 时长 | 30秒 | 预热充分+持续观察+不浪费 |
| 库存 | 1000件 | 5000请求vs1000库存，验证超卖 |

#### 四、为什么能达到5000 QPS？（必问）

这是面试官最关心的问题。我来解释一下架构设计：

```
┌─────────────────────────────────────────────────────────────────┐
│                  5000 QPS架构解析                               │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  5000请求/秒                                                     │
│       │                                                         │
│       ▼                                                         │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  第1层：Redis Lua脚本  （拦截99%流量）                   │   │
│  │  - 一人一单检查：SISMEMBER voucher:order:1 userId       │   │
│  │  - 库存扣减：DECR voucher:stock:1                        │   │
│  │  - 原子操作：Lua脚本保证原子性                           │   │
│  │  - 耗时：~5ms                                            │   │
│  │  - 4950个请求在这里被拦截（库存不足/已购买）             │   │
│  └────────────────────────┬────────────────────────────────┘   │
│                           │ 50个请求通过                        │
│                           ▼                                    │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  第2层：Kafka异步写入  （解耦+削峰）                     │   │
│  │  - 发送消息到order-create topic                          │   │
│  │  - 立即返回，不等待数据库                                │   │
│  │  - 耗时：<1ms（内存操作）                               │   │
│  │  - 50个订单消息进入Kafka队列                            │   │
│  └────────────────────────┬────────────────────────────────┘   │
│                           │ 异步消费                           │
│                           ▼                                    │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  第3层：Kafka消费者  （数据库操作）                      │   │
│  │  - 乐观锁扣减库存：UPDATE ... WHERE stock > 0           │   │
│  │  - 创建订单记录                                          │   │
│  │  - 异步处理，不阻塞主流程                                │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  总结：99%的请求在Redis层处理，只有1%需要异步处理数据库         │
│  → 即使数据库性能一般，也能扛5000+ QPS                         │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**为什么快？三个关键点：**

1. **Redis拦截流量**：99%请求在Redis层被拦截，不打数据库
2. **Lua原子操作**：检查+扣减原子完成，无锁竞争
3. **Kafka异步**：主流程不等待数据库，立即返回
