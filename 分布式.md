## 1. 项目中实现的分布式锁

### 1.1 核心场景说明

在分布式系统里，多个服务节点要操作共享资源，比如商品库存扣减、订单号生成，本地锁只能管单个 JVM 里的线程，跨节点就没用了，容易出现超卖、重复数据这些问题。而 Redis 性能高、支持原子命令，所以成为分布式锁的首选，能高效解决跨节点互斥的需求。

### 1.2 项目中的实现

#### 1.2.1 优化后的Redis分布式锁实现

项目优化后的分布式锁实现位于 `shop-service/utils/cache_client.go`：

```go
// 生成唯一锁值
func (c *CacheClient) generateLockValue() string {
    return uuid.New().String()
}

// TryLock 尝试获取分布式锁
func (c *CacheClient) TryLock(ctx context.Context, key string) bool {
    lockValue := c.generateLockValue()
    ok, err := c.rdb.SetNX(ctx, key, lockValue, time.Duration(LockShopTTL)*time.Second).Result()
    if err != nil || !ok {
        return false
    }
    // 保存本地映射，用于释放锁时校验
    c.localMu.Lock()
    c.localLockValue[key] = lockValue
    c.localMu.Unlock()
    return true
}
```

锁的key定义在 `shop-service/utils/redis_constants.go`：

```go
const (
    LockShopKey = "lock:shop:"
    LockShopTTL = 30 // 生产环境推荐30秒
)
```

#### 1.2.2 安全释放锁（使用Lua脚本）

```go
// Lua脚本：先判断value是否匹配，再删除
var safeUnlockScript = redis.NewScript(`
    if redis.call('get', KEYS[1]) == ARGV[1] then
        return redis.call('del', KEYS[1])
    else
        return 0
    end
`)

// SafeUnlock 安全释放分布式锁
func (c *CacheClient) SafeUnlock(ctx context.Context, key string) error {
    c.localMu.RLock()
    lockValue, exists := c.localLockValue[key]
    c.localMu.RUnlock()

    if !exists {
        return nil
    }
    // 使用Lua脚本原子执行：先判断再删除
    _, err := safeUnlockScript.Run(ctx, c.rdb, []string{key}, lockValue).Result()

    c.localMu.Lock()
    delete(c.localLockValue, key)
    c.localMu.Unlock()
    return err
}
```

#### 1.2.4 看门狗机制（自动续期）

```go
// WatchdogLock 加锁并启动看门狗
func (c *CacheClient) WatchdogLock(ctx context.Context, key string) bool {
    lockValue := c.generateLockValue()
    ok := c.ReentrantLockWithValueAndTTL(ctx, key, lockValue, 30*time.Second)
    if !ok {
        return false
    }
    // 启动看门狗协程
    go c.runWatchdog(ctx, key, lockValue)
    return true
}

// 看门狗：每隔1/3TTL检查并续期
func (c *CacheClient) runWatchdog(ctx context.Context, key, lockValue string) {
    watchdogInterval := 10 * time.Second // 30秒TTL的1/3
    ticker := time.NewTicker(watchdogInterval)
    defer ticker.Stop()

    for {
        select {
        case <-ticker.C:
            c.localMu.RLock()
            _, exists := c.localLockValue[key]
            c.localMu.RUnlock()
            if !exists {
                return
            }
            currentValue, _ := c.rdb.Get(ctx, key).Result()
            if currentValue != lockValue {
                return
            }
            // 续期30秒
            _ = c.rdb.Expire(ctx, key, 30*time.Second)
        case <-ctx.Done():
            return
        }
    }
}
```

### 1.3 面试标准回答（深度版）

#### 1.3.1 基础实现

**问题：Redis分布式锁如何实现？**

**核心：保证加锁和释放锁的原子性。**

**1. 加锁实现**

项目使用 Redis 的 `SET key value NX EX` 命令（SetNX），原子完成"不存在则设置"和"过期时间"两个操作。

**为什么不能用 SETNX + EXPIRE 两步走？**

两步骤非原子，存在进程崩溃死锁风险。Redis 2.6.12 后使用单条命令。

**Key 和 Value 设计**：

- **Key**：`lock:{业务}:{资源ID}`
- **Value**：使用 UUID，用于校验锁归属

#### 1.3.2 释放锁的问题

**问题**：直接 DEL 可能误删别人持有的锁。

**场景**：

```
1. 线程A获取锁（TTL=30s），执行业务（耗时35s）
2. 30秒后锁过期释放
3. 线程B获取锁成功
4. 线程A业务执行完，执行DEL
5. 结果：线程B的锁被A删除了！
```

**解决方案：Lua脚本原子操作**

```lua
if redis.call('get', KEYS[1]) == ARGV[1] then
    return redis.call('del', KEYS[1])
else
    return 0
end
```

#### 1.3.3 锁超时问题

项目 TTL = 30秒，使用看门狗机制自动续期。

**看门狗工作流程**：

```
1. 加锁成功，启动看门狗
2. 每10秒检查一次
3. 如果仍持有锁，TTL重置为30秒
4. 业务执行完毕，释放锁，看门狗停止
5. 客户端崩溃 -> 看门狗停止 -> 30秒后锁过期
```

#### 1.3.4 可重入锁

使用 Hash 结构：

```
Key: lock:shop:123
Hash:
  field: uuid-clientID = 重入次数
```

#### 1.3.5 Redis集群问题

**脑裂问题**：主从切换时锁丢失。

**解决方案**：RedLock、Zookeeper。

## 2. 针对目前项目的高频面试追问

### 2.1 基础问题

#### 问题1：SETNX + EXPIRE 两步操作有什么风险？

1. **进程崩溃**：SETNX成功、EXPIRE执行前崩溃，锁永不过期
2. **网络问题**：SETNX成功但EXPIRE请求未送达

---

#### 问题2：释放锁时为什么要先检查再删除？

直接 DEL 会误删别人持有的锁。

**解决方案**：Lua脚本原子操作。

---

#### 问题3：Redis 主从架构下分布式锁有什么问题？

**脑裂问题**：主节点加锁成功后未同步到从节点就宕机。

---

#### 问题4：什么是看门狗机制？

自动续期机制，解决业务耗时超过 TTL 的问题。

---

#### 问题5：分布式锁可以重入吗？如何实现？

可以。使用 Hash 结构存储重入次数。

---

#### 问题6：缓存击穿和缓存穿透有什么区别？

| | 缓存击穿 | 缓存穿透 |
|--|----------|----------|
| 定义 | 热点key过期，大量请求涌入DB | 查询不存在的数据，直接打到DB |
| 解决方案 | 互斥锁 | 布隆过滤器、空值缓存 |

#### 问题9：如何解决超卖问题？

**多层防护**：

1. **Redis 原子扣减**：Lua脚本

3. **唯一索引**：防止重复订单

```lua

if not stock or tonumber(stock) <= 0 then
    return -1
end
return redis.call('decr', KEYS[1])
```
